## Test Structure

<Warning>
  **Important**: Test files must be placed in `__tests__` directories at the
  package/app root level, NOT inside `src/` directories. This is a monorepo-wide
  standard.
</Warning>

### Directory Organization

```
package-or-app/
├── __tests__/              # Tests at root, NOT in src/
│   ├── setup.ts           # Test setup file
│   ├── unit/              # Unit tests
│   ├── integration/       # Integration tests
│   └── e2e/              # E2E tests (*.spec.ts)
├── src/
│   └── components/
│       ├── HelloWorld.tsx
│       └── HelloWorld.test.tsx  # Colocated test (optional)
├── vitest.config.ts       # Vitest config
└── playwright.config.ts   # Playwright config (if E2E tests exist)
```

### File Naming Conventions

- **Unit/Integration Tests**: `*.test.{ts,tsx}` (for Vitest)
- **E2E Tests**: `*.spec.ts` (for Playwright)
- **Test Files Location**:
  - **Preferred**: `__tests__/` directory at package root
  - **Allowed**: Colocated tests next to components (e.g., `Button.test.tsx`
    next to `Button.tsx`)

### Test Configuration

All test configurations are provided by the `@repo/qa` package:

```typescript
// vitest.config.ts
import { createNextAppConfig } from "@repo/qa/vitest/configs";

export default createNextAppConfig({
  setupFiles: ["./__tests__/setup.ts"]
});

// playwright.config.ts
import { createAppPlaywrightConfig } from "@repo/qa/playwright";

export default createAppPlaywrightConfig({
  name: "app-name",
  port: 3000,
  baseURL: "http://localhost:3000"
});
```

## Data-TestId Standards

### Overview

Data-testid attributes provide stable, maintainable testing selectors that won't
break when styling or content changes. These standards ensure consistent,
reliable test selectors across all components.

<CodeGroup>
```tsx Naming Convention
// General pattern
data-testid="[component-name]-[element-type]-[modifier]"

// Examples data-testid="product-card-root" data-testid="add-to-cart-button"
data-testid="navigation-menu-mobile" data-testid="switch-input-active"

````

```tsx Component Interface
interface ComponentProps {
  testId?: string;
  buttonTestId?: string;
  badgeTestId?: string;
  iconTestId?: string;
}

export function AddToCardButton({
  testId = 'add-to-cart',
  buttonTestId = 'add-to-cart-button',
  badgeTestId = 'add-to-cart-badge',
  iconTestId = 'add-to-cart-icon',
  ...props
}: ComponentProps) {
  return (
    <div data-testid={testId}>
      <Button data-testid={buttonTestId}>
        <ShoppingBagIcon data-testid={iconTestId} />
        {count > 0 && <Badge data-testid={badgeTestId}>{count}</Badge>}
      </Button>
    </div>
  );
}
````

</CodeGroup>

### Testing Guidelines

<Tabs>
  <Tab title="Best Practices">
    **Recommended Patterns:**
    - Use kebab-case for all data-testid values
    - Include component context in the prefix
    - Add state modifiers when relevant (`-active`, `-disabled`)
    - Provide testId props for flexibility
    - Test behavior, not implementation details

    **Performance Considerations:**
    - Use specific testids to reduce DOM traversal
    - Pre-calculate testids when possible
    - Keep testid values concise

  </Tab>

  <Tab title="Testing Examples">
    ```tsx
    // Good practice - using data-testid
    test('should add item to cart', () => {
      render(<AddToCardButton />);

      const button = screen.getByTestId('add-to-cart-button');
      fireEvent.click(button);

      const badge = screen.getByTestId('add-to-cart-badge');
      expect(badge).toHaveTextContent('1');
    });

    // Playwright/Cypress usage
    await page.getByTestId('add-to-cart-button').click();
    cy.get('[data-testid="add-to-cart-button"]').click();
    ```

  </Tab>
</Tabs>

    export async function waitForElement(
      getElement: () => HTMLElement | null,
      timeout = 5000
    ): Promise<HTMLElement> {
      const start = Date.now();

      while (Date.now() - start < timeout) {
        const element = getElement();
        if (element) return element;

        await new Promise(resolve => setTimeout(resolve, 100));
      }

      timeout = 5000
    ): Promise<void> {

        if (condition()) return;
        await new Promise(resolve => setTimeout(resolve, 100));
      }

      throw new Error('Condition not met within timeout');
    }

## Best Practices

<Warning>
  **Testing Guidelines:** - Write tests before implementing features (TDD) - Use
  descriptive test names that explain the expected behavior - Test user
  interactions, not implementation details - Mock external dependencies and
  services - Maintain high test coverage (>80%) for critical code paths - Use
  data-testid for reliable element selection
</Warning>

### Recommended Patterns

1. **Test Organization**
   - Group related tests in describe blocks
   - Use consistent naming conventions
   - Keep test files close to source files

2. **Mocking Strategy**
   - Mock at the boundary level (API calls, database operations)
   - Avoid mocking internal implementation details
   - Use factories for test data generation

3. **Performance**
   - Use appropriate test environments (jsdom vs node)
   - Clean up resources in afterEach/afterAll
   - Avoid unnecessary re-renders in tests

4. **Maintainability**
   - Keep tests simple and focused
   - Use shared utilities for common operations
   - Update tests when requirements change

The testing framework provides comprehensive tools for building reliable,
maintainable test suites that ensure code quality and prevent regressions across
the entire Forge platform.
