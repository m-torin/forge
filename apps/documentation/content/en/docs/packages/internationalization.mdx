# Internationalization Package

Server-side internationalization system with **AI-powered translations**, **type-safe
dictionaries**, and **Next.js middleware** for building multilingual applications.

## Overview

The internationalization package provides a lightweight, server-focused i18n solution optimized for
Next.js App Router:

- **Server-Side Only**: No client bundles, perfect for SSR/SSG applications
- **Type-Safe Dictionaries**: Full TypeScript support with autocomplete for translation keys
- **AI Translations**: Automated translations via Languine and DeepSeek API
- **Middleware Routing**: Automatic locale detection and URL management
- **6 Languages**: English, Spanish, German, French, Portuguese, and Chinese
- **Zero Client Runtime**: All translations resolved at build/request time

## Core Concepts

### 1. Server-Side Architecture

This package is designed exclusively for server-side usage:

```typescript
// ✅ Server Component (Recommended)
import { getDictionary } from '@repo/internationalization';

export default async function Page({ params: { locale } }) {
  const dict = await getDictionary(locale);
  return <h1>{dict.web.home.hero.title}</h1>;
}

// ❌ Client Component (Not Supported)
'use client';
// No useTranslation or similar hooks available
```

### 2. Dictionary Structure

Dictionaries are organized hierarchically in JSON files:

```json
// dictionaries/en.json
{
  "common": {
    "brand": "Forge",
    "navigation": {
      "home": "Home",
      "about": "About",
      "contact": "Contact"
    }
  },
  "web": {
    "home": {
      "hero": {
        "title": "Welcome to Forge",
        "subtitle": "Build better products faster"
      },
      "features": [
        {
          "title": "Fast",
          "description": "Lightning fast performance"
        }
      ]
    }
  }
}
```

### 3. Type Safety

Full TypeScript support with auto-completion:

```typescript
import { getDictionary, type Dictionary } from '@repo/internationalization';

// Type-safe dictionary access
const dict = await getDictionary('en');
dict.web.home.hero.title; // ✅ Auto-completed
dict.web.home.invalid; // ❌ TypeScript error

// Export dictionary type for components
type Props = {
  dict: Dictionary;
};
```

## Basic Usage

### Server Components

```typescript
import { getDictionary } from '@repo/internationalization';

export default async function HomePage({
  params: { locale }
}: {
  params: { locale: string }
}) {
  const dict = await getDictionary(locale);

  return (
    <main>
      <h1>{dict.web.home.hero.title}</h1>
      <p>{dict.web.home.hero.subtitle}</p>

      {/* Array translations */}
      {dict.web.home.features.map((feature, index) => (
        <div key={index}>
          <h3>{feature.title}</h3>
          <p>{feature.description}</p>
        </div>
      ))}
    </main>
  );
}
```

### Layouts

```typescript
import { getDictionary } from '@repo/internationalization';

export default async function RootLayout({
  children,
  params: { locale },
}: {
  children: React.ReactNode;
  params: { locale: string };
}) {
  const dict = await getDictionary(locale);

  return (
    <html lang={locale}>
      <body>
        <nav>
          {dict.common.navigation.home}
        </nav>
        {children}
      </body>
    </html>
  );
}
```

### Passing to Client Components

Since there are no client hooks, pass translations as props:

```typescript
// Server Component
import { getDictionary } from '@repo/internationalization';
import ClientButton from './ClientButton';

export default async function Page({ params: { locale } }) {
  const dict = await getDictionary(locale);

  return (
    <ClientButton
      text={dict.common.buttons.submit}
      cancelText={dict.common.buttons.cancel}
    />
  );
}

// Client Component
'use client';

export default function ClientButton({
  text,
  cancelText
}: {
  text: string;
  cancelText: string;
}) {
  return (
    <>
      <button>{text}</button>
      <button>{cancelText}</button>
    </>
  );
}
```

## Middleware Configuration

### Basic Setup

```typescript
// middleware.ts
import { internationalizationMiddleware } from '@repo/internationalization/middleware';

export function middleware(request: Request) {
  return internationalizationMiddleware(request);
}

export const config = {
  matcher: [
    // Skip API routes and static files
    '/((?!api|_next/static|_next/image|favicon.ico).*)',
  ],
};
```

### How It Works

The middleware:

1. **Detects user locale** from `Accept-Language` headers
2. **Matches against supported locales** using W3C standards
3. **Rewrites URLs** (e.g., `/about` → `/es/about` for Spanish users)
4. **Handles default locale** (English has no prefix: `/` not `/en/`)

### Custom Locale Detection

```typescript
import { createInternationalizationMiddleware } from '@repo/internationalization/middleware';

const middleware = createInternationalizationMiddleware({
  locales: ['en', 'es', 'de'],
  defaultLocale: 'en',
  // Custom locale detection logic
  getLocale: (request) => {
    // Check cookie first
    const cookieLocale = request.cookies.get('locale');
    if (cookieLocale) return cookieLocale;

    // Fall back to header detection
    return detectLocaleFromHeaders(request);
  },
});
```

## AI-Powered Translations

### Languine Integration

The package uses Languine for automated translations:

```bash
# Translate all dictionaries
pnpm translate

# Update specific language
pnpm translate --lang es

# Force retranslation
pnpm translate --force
```

### Configuration

```json
// languine.json
{
  "version": "1.0.0",
  "locale": {
    "source": "en",
    "targets": ["es", "de", "fr", "pt", "zh"]
  },
  "files": {
    "include": ["dictionaries/en.json"]
  },
  "llm": {
    "provider": "deepseek",
    "model": "deepseek-chat",
    "temperature": 0
  }
}
```

### Translation Locking

Translations are locked to ensure consistency:

```json
// languine.lock
{
  "translations": {
    "common.brand": {
      "es": "Forge",
      "de": "Forge"
      // Brand names preserved
    },
    "web.home.hero.title": {
      "es": "Bienvenido a Forge",
      "de": "Willkommen bei Forge"
      // Translated content
    }
  }
}
```

## Extending Dictionaries

Apps can extend the base dictionary with custom translations:

```json
// app/i18n/dictionaries/en.json
{
  "app": {
    "dashboard": {
      "title": "Dashboard",
      "widgets": {
        "sales": "Sales Overview"
      }
    },
    "l": {
      "en": "English",
      "fr": "Français",
      "es": "Español"
    }
  }
}
```

```typescript
// app/i18n/index.ts
import { createDictionary } from '@repo/internationalization/extend';
import { getDictionary as getBaseDictionary } from '@repo/internationalization';

async function getAppDictionary(locale: string) {
  const dict = await import(`./dictionaries/${locale}.json`);
  return dict.default;
}

export const getDictionary = createDictionary(getBaseDictionary, getAppDictionary);

// Usage
const dict = await getDictionary('en');
dict.web.home.hero.title; // From base dictionary
dict.app.dashboard.title; // From app dictionary
```

## Advanced Patterns

### Dynamic Locale Switching

Since there are no client hooks, implement locale switching via navigation:

```typescript
// LocaleSwitcher.tsx
import { Link } from '@repo/internationalization/client';
import { locales } from '@repo/internationalization';

export default function LocaleSwitcher({
  currentLocale
}: {
  currentLocale: string
}) {
  return (
    <select
      value={currentLocale}
      onChange={(e) => {
        // Navigate to same page with different locale
        window.location.pathname = `/${e.target.value}${
          window.location.pathname.replace(/^\/[a-z]{2}/, '')
        }`;
      }}
    >
      {locales.map((locale) => (
        <option key={locale} value={locale}>
          {locale.toUpperCase()}
        </option>
      ))}
    </select>
  );
}
```

## Best Practices

### 1. Dictionary Organization

- **Group by feature**: Keep related translations together
- **Use consistent naming**: Follow a clear hierarchy
- **Avoid deep nesting**: Maximum 3-4 levels deep
- **Array for lists**: Use arrays for repeated content
- **Use concise keys**: Prefer `l` for language names instead of `languages`

Example of recommended structure:

```json
{
  "app": {
    "l": {
      "en": "English",
      "fr": "Français",
      "es": "Español",
      "de": "Deutsch",
      "pt": "Português"
    }
  }
}
```

### 2. Performance

- **Server-only imports**: Never import dictionaries in client components
- **Memoize in layouts**: Dictionary calls are cached per request
- **Avoid runtime parsing**: All translations resolved at build time

### 3. Type Safety

```typescript
// Export specific dictionary sections
export type NavigationDict = Dictionary['common']['navigation'];

// Component props
type Props = {
  navigation: NavigationDict;
};
```

### 4. Translation Keys

```typescript
// ❌ Bad: Dynamic keys
const key = `web.home.features.${index}.title`;
const translation = dict[key]; // No type safety

// ✅ Good: Direct access
const translation = dict.web.home.features[index].title;
```

## Limitations

1. **No Client Hooks**: All translations must be passed as props to client components
2. **No Dynamic Formatting**: No built-in pluralization or date/number formatting
3. **No Namespacing**: All translations in a single dictionary per locale
4. **Static Only**: No runtime translation loading or hot-reloading
5. **No Context API**: Cannot access translations via React context

## Migration Guide

### From Client-Side i18n

```typescript
// Before (with react-i18next)
'use client';
import { useTranslation } from 'react-i18next';

function Component() {
  const { t } = useTranslation();
  return <h1>{t('home.title')}</h1>;

// After (server component)
import { getDictionary } from '@repo/internationalization';

async function Component({ locale }) {
  const dict = await getDictionary(locale);
  return <h1>{dict.home.title}</h1>;
}
```
