# Concepts

Core concepts and patterns used throughout the Forge monorepo.

## Overview

This section explains the common patterns, conventions, and file structures you'll encounter across
the monorepo. Understanding these concepts will help you navigate and contribute to the codebase
effectively.

## Key Files and Patterns

### keys.ts - Environment Variable Management

Every package and app that requires environment variables has a `keys.ts` file that:

- Validates environment variables using `@t3-oss/env-nextjs`
- Provides type-safe access to configuration
- Handles different environments (development, production, CI/CD)
- Supports graceful degradation when variables are optional

#### Example Structure

```typescript
import { createEnv } from '@t3-oss/env-nextjs';
import { z } from 'zod';

export const keys = () =>
  createEnv({
    server: {
      API_KEY: z.string().min(1),
      OPTIONAL_KEY: z.string().optional(),
    },
    client: {
      NEXT_PUBLIC_APP_URL: z.string().url(),
    },
    runtimeEnv: {
      API_KEY: process.env.API_KEY,
      OPTIONAL_KEY: process.env.OPTIONAL_KEY,
      NEXT_PUBLIC_APP_URL: process.env.NEXT_PUBLIC_APP_URL,
    },
  });
```

#### Key Principles

1. **Lazy Evaluation**: Keys are exported as functions to allow graceful initialization
2. **Type Safety**: Zod schemas ensure runtime validation
3. **Environment Awareness**: Different validation rules for development vs production
4. **No ReturnType**: Functions should not use explicit ReturnType annotations

### env.ts - Application Environment Configuration

Located in the root of each app (never in `src/`), the `env.ts` file:

- Centralizes all environment variables for the application
- Re-exports keys from used packages
- Adds app-specific environment variables
- Provides a single source of truth for configuration

#### Example Structure

```typescript
import { keys as analyticsKeys } from '@repo/analytics/keys';
import { keys as authKeys } from '@repo/auth/keys';
import { createEnv } from '@t3-oss/env-nextjs';
import { z } from 'zod';

export const env = createEnv({
  extends: [analyticsKeys(), authKeys()],
  server: {
    APP_SPECIFIC_KEY: z.string(),
  },
  client: {
    NEXT_PUBLIC_FEATURE_FLAG: z.boolean(),
  },
  runtimeEnv: {
    APP_SPECIFIC_KEY: process.env.APP_SPECIFIC_KEY,
    NEXT_PUBLIC_FEATURE_FLAG: process.env.NEXT_PUBLIC_FEATURE_FLAG === 'true',
  },
});
```

### Lazy Initialization Pattern

Many packages use lazy initialization to avoid errors when environment variables aren't configured:

```typescript
let instance: ServiceClient | null = null;

export function getService(): ServiceClient {
  if (!instance) {
    const env = keys();
    if (!env.SERVICE_API_KEY) {
      // Return mock in development
      return new MockServiceClient();
    }
    instance = new ServiceClient(env.SERVICE_API_KEY);
  }
  return instance;
}
```

### Provider Pattern

Services that require configuration use the provider pattern:

```typescript
// 1. Define provider interface
interface StorageProvider {
  upload(key: string, data: Buffer): Promise<void>;
  download(key: string): Promise<Buffer>;
}

// 2. Implement providers
class S3Provider implements StorageProvider {
  /* ... */
}
class LocalProvider implements StorageProvider {
  /* ... */
}

// 3. Factory function
export function createStorageProvider(config: Config): StorageProvider {
  switch (config.type) {
    case 's3':
      return new S3Provider(config);
    case 'local':
      return new LocalProvider(config);
    default:
      throw new Error(`Unknown provider: ${config.type}`);
  }
}
```

### Mock Patterns for Development

Packages provide mock implementations when not configured:

```typescript
class MockEmailService {
  async send(email: EmailOptions) {
    console.log('📧 Mock email sent:', email);
    return { id: 'mock-id', status: 'sent' };
  }
}

export const email = process.env.RESEND_API_KEY ? new ResendEmailService() : new MockEmailService();
```

### Type-Safe Error Handling

Consistent error handling across packages:

```typescript
export class PackageError extends Error {
  constructor(
    message: string,
    public code: string,
    public statusCode?: number
  ) {
    super(message);
    this.name = 'PackageError';
  }
}

// Usage
throw new PackageError('Invalid API key', 'INVALID_API_KEY', 401);
```

### Middleware Composition

Packages that provide middleware use composition:

```typescript
export function createMiddleware(options?: Options) {
  return async function middleware(request: Request, context: Context, next: NextFunction) {
    // Pre-processing
    const result = await processRequest(request, options);

    // Call next middleware
    const response = await next(request, context);

    // Post-processing
    return processResponse(response, result);
  };
}
```

### Server/Client Separation

Packages clearly separate server and client code:

```
package/
├── index.ts        # Client-side exports
├── server.ts       # Server-side exports
├── client.ts       # Client-only code
├── keys.ts         # Environment variables
└── types.ts        # Shared types
```

### Configuration Inheritance

Shared configurations use extension patterns:

```typescript
// Base configuration
export const baseConfig = {
  timeout: 5000,
  retries: 3,
};

// Extended configuration
export const productionConfig = {
  ...baseConfig,
  timeout: 30000,
  retries: 5,
};
```

## Common Conventions

### Naming Conventions

- **Files**: `kebab-case.ts` for files, except for React components
- **Components**: `PascalCase.tsx` for React components
- **Exports**: Named exports preferred over default exports
- **Types**: `PascalCase` for types and interfaces
- **Constants**: `SCREAMING_SNAKE_CASE` for constants

### Import Organization

Standard import order across all files:

```typescript
// 1. React/Next.js imports
import { useState } from 'react';
import { useRouter } from 'next/navigation';

// 2. External packages
import { z } from 'zod';
import { format } from 'date-fns';

// 3. Internal packages (@repo/*)
import { auth } from '@repo/auth';
import { database } from '@repo/database';

// 4. Local imports
import { Button } from './components/button';
import type { User } from './types';
```

### Testing Patterns

Tests follow consistent patterns:

```typescript
// 1. Test file naming
component.test.tsx; // Unit tests
component.e2e.test.tsx; // E2E tests

// 2. Test structure
describe('ComponentName', () => {
  beforeEach(() => {
    // Setup
  });

  it('should handle normal case', () => {
    // Test implementation
  });

  it('should handle error case', () => {
    // Test error handling
  });
});

// 3. Mock location
__mocks__ / module - name.ts;
```

### Documentation Standards

Every package includes:

1. **README.md**: Overview and basic usage
2. **JSDoc comments**: For public APIs
3. **Type definitions**: Exported types with descriptions
4. **Examples**: Common usage patterns

### Error Messages

Consistent error message format:

```typescript
// Good
throw new Error(`Failed to connect to database: ${error.message}`);

// Better
throw new DatabaseError(`Failed to connect to database at ${host}:${port}`, 'CONNECTION_FAILED', {
  host,
  port,
  originalError: error,
});
```

## Package Structure

Standard package structure:

```
@repo/package-name/
├── __tests__/          # Test files
├── src/                # Source code (if used)
├── index.ts            # Main exports
├── server.ts           # Server-side exports
├── client.ts           # Client-side exports
├── keys.ts             # Environment variables
├── types.ts            # TypeScript types
├── package.json        # Package manifest
├── tsconfig.json       # TypeScript config
├── vitest.config.ts    # Test configuration
└── README.md           # Documentation
```

## Best Practices

### 1. Environment Variables

- Always validate with Zod schemas
- Provide helpful error messages for missing variables
- Use optional() for non-critical variables
- Group related variables together

### 2. Type Safety

- Export types from types.ts
- Use discriminated unions for variants
- Avoid `any` types (use `unknown` if needed)
- Document complex types with JSDoc

### 3. Error Handling

- Create specific error classes
- Include error codes for programmatic handling
- Preserve stack traces
- Log errors appropriately

### 4. Performance

- Use lazy initialization
- Implement caching where appropriate
- Avoid synchronous I/O in hot paths
- Monitor with observability tools

### 5. Security

- Never log sensitive data
- Validate all inputs
- Use environment variables for secrets
- Implement rate limiting
