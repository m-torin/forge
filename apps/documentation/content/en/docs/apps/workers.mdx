## Authentication

All API endpoints require authentication via API key or session cookie.

### Authentication Methods

#### 1. User API Keys

Generated through the Better Auth system:

1. Sign in to the Workers app at `/sign-in`
2. Navigate to your account settings
3. Create a new API key with appropriate permissions
4. Copy the key immediately (it won't be shown again)

#### 2. Service API Key

For service-to-service authentication and CI/CD:

- Set via `SERVICE_API_KEY` environment variable
- Minimum 32 characters for security
- Can be rotated via Doppler without code changes
- Provides system-level access without user context

### Using API Keys

```bash
# Using user API key
curl -X POST https://workers.app/api/client/trigger \
  -H "Content-Type: application/json" \
  -H "x-api-key: user_xxxxxxxxxxxxxxxx" \
  -d '{"url": "...", "body": {...}}'

# Using service API key (for CI/CD)
curl -X POST https://workers.app/api/workflows/basic \
  -H "x-api-key: YOUR_SERVICE_API_KEY" \
  -d '{"data": "..."}'

# Using session cookie (after sign-in)
curl -X POST https://workers.app/api/workflows/basic \
  -H "Cookie: session=..." \
  -d '{"data": "..."}'
```

### API Key Permissions

User API keys can have scoped permissions:

- `workflows:read` - View workflow status
- `workflows:write` - Trigger and cancel workflows
- `workflows:admin` - Manage workflow configurations
- `observability:read` - Access monitoring data

### Security Best Practices

1. **Rotate Keys Regularly** - Use Doppler for easy rotation
2. **Use Environment Variables** - Never hardcode keys
3. **Scope Permissions** - Grant minimum required access
4. **Monitor Usage** - Track API key usage in observability dashboard

### Rate Limits

Different endpoints have different rate limits:

| Endpoint Type | Limit        | Window     |
| ------------- | ------------ | ---------- |
| Default       | 60 requests  | 60 seconds |
| Workflows     | 10 requests  | 60 seconds |
| Trigger       | 20 requests  | 60 seconds |
| Observability | 100 requests | 60 seconds |

## Workflow Development

### Creating a New Workflow

```typescript
// app/workflows/my-workflow/definition.ts
import { workflow } from '@upstash/workflow';

export const myWorkflow = workflow('my-workflow')
  .step('validate', async ({ input }) => {
    // Validation logic
    return { valid: true, data: input };
  })
  .step('process', async ({ data }) => {
    // Main processing
    return { result: processedData };
  })
  .step('notify', async ({ result }) => {
    // Send notifications
    await sendNotification(result);
    return { success: true };
  });
```

### Registering Workflows

```typescript
// app/workflows/registry.ts
import { myWorkflow } from './my-workflow/definition';

export const workflows = {
  'my-workflow': {
    workflow: myWorkflow,
    metadata: {
      name: 'My Workflow',
      description: 'Process data with custom logic',
      category: 'processing',
    },
  },
};
```

## Security

### Authentication Flow

1. All routes except `/api/auth` require authentication
2. Service API keys checked first for system access
3. User API keys or sessions checked second
4. Webhook endpoints use signature verification

### Webhook Verification

QStash webhooks are verified using signature:

```typescript
// Automatic verification for QStash webhooks
import { verifySignatureAppRouter } from '@upstash/qstash/nextjs';

export async function POST(request: Request) {
  // Signature automatically verified
  await verifySignatureAppRouter(request);

  // Process webhook
  const data = await request.json();
}
```

## Deployment

### Pre-Deployment Checklist

- [ ] Remove or conditionally exclude local development settings
- [ ] Ensure all environment variables are configured
- [ ] Test workflows locally with production-like settings
- [ ] Review security implementations

### Environment Variables

Required environment variables for production:

```bash
# Authentication
SERVICE_API_KEY=... # Minimum 32 characters
BETTER_AUTH_SECRET=...
DATABASE_URL=...

# QStash
QSTASH_TOKEN=...
QSTASH_CURRENT_SIGNING_KEY=...
QSTASH_URL=https://qstash.upstash.io # Remove for local dev

# Redis
UPSTASH_REDIS_REST_URL=...
UPSTASH_REDIS_REST_TOKEN=...

# NextAuth
NEXTAUTH_URL=https://workers.yourdomain.com
NEXT_PUBLIC_APP_URL=https://workers.yourdomain.com

# Optional
SKIP_WORKFLOW_DEDUPLICATION=false
WORKFLOW_DEV_MODE=false
TRUSTED_IPS=ip1,ip2,ip3 # For additional security
```

### Variables to Remove in Production

- `QSTASH_URL` - Only used for local development with QStash mock
- `UPSTASH_WORKFLOW_URL` - Only used with local tunnels

## Common Use Cases

### 1. Batch Product Import

```typescript
// Trigger batch import
await fetch('/api/workflows/product-import', {
  method: 'POST',
  headers: {
    'x-api-key': API_KEY,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    csvUrl: 'https://...',
    mappings: {...},
  }),
});
```

### 2. Scheduled Reports

```typescript
// Schedule daily report
await fetch('/api/schedules', {
  method: 'POST',
  headers: { 'x-api-key': API_KEY },
  body: JSON.stringify({
    workflow: 'generate-report',
    schedule: '0 9 * * *', // 9 AM daily
    data: { reportType: 'sales' },
  }),
});
```

### 3. Event-Driven Processing

```typescript
// Process on webhook
export async function POST(request: Request) {
  const event = await request.json();

  // Trigger workflow
  await triggerWorkflow('process-event', {
    eventType: event.type,
    payload: event.data,
  });
}
```
